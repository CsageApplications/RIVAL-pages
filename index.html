How It Works

Agent writes code. R!VAL decides.

Your agent gets a spec. It generates code. R!VAL checks every file the agent produces against that spec — five domains in parallel — before anything moves forward. No trust. Just proof.

01
Define the spec
Capture intent as an R!Spec contract — acceptance criteria, constraints, failure modes. This is the source of truth the agent is held against. Vague language is rejected before the agent ever runs.
02
Agent generates code
Your AI agent — Cursor, Copilot, Claude, Devin, or your own — produces an implementation. R!VAL treats every file it writes as untrusted output until proven otherwise.
03
R!VAL validates the output
Five domains run in parallel against the agent's code: did it follow the spec? Did it hallucinate scope? Does it introduce security risk? Is it trying to do something it wasn't told to do?
04
PASS / WARN / BLOCK
A governed decision tied directly to the spec the agent was given. BLOCK is not a suggestion — it means the agent's output does not advance until the violation is resolved.

Validation Domains

Five ways an agent can fail you.

Agents don't just write buggy code — they hallucinate features, expand scope, skip security constraints, and ignore failure modes. R!VAL checks all five failure surfaces on every file, every time.

Spec Integrity
Verifies the spec itself is complete, measurable, and traceable before any code is touched. Rejects ambiguous language before the pipeline runs.
Ambiguous language detection ("fast", "good", "scalable")
Acceptance criteria completeness and measurability
Failure mode coverage — edge cases and degradation paths
Constraint presence and specificity
Agent Behavior
The domain built specifically for AI-generated code. Detects when an agent goes off-script — hallucinating components, expanding scope beyond the spec, or making autonomous decisions it was never authorized to make.
Hallucination detection — components not in the spec
Scope creep — features added beyond spec boundaries
Unsafe autonomous decisions without spec authorization
Dependency drift — linking to undeclared services
Security Risk
OWASP Top 10 checks embedded in every validation run — before the code is committed, not after it's been in production for six months.
Auth and authorization pattern validation
Secrets and credential exposure detection
Injection vulnerability surface analysis
Insecure data handling and storage patterns
Engineering Quality
Scores the structural and operational quality of the implementation against the spec's declared requirements — test coverage, error handling, and maintainability.
Test coverage adequacy against specified criteria
Error handling completeness for declared failure modes
Performance constraint compliance
Code maintainability and documentation standards
Policy Governance
Enforces organization-specific rules, regulatory requirements, and architectural decisions that go beyond individual spec quality.
GDPR, SOC2, HIPAA compliance pattern checks
Org-specific architecture and tech-stack rules
Data residency and sovereignty constraints
Licensing and dependency policy enforcement
Enforcement

Four layers.
Zero bypass.

An agent that can skip validation is an agent you can't govern. R!VAL is embedded at every layer — CLI, file watcher, pre-commit, and CI gate — so there is no gap an agent can write through undetected.

1
CLI — Per-file Validation
After every file the AI writes, the agent must run R!VAL. Exit code 1 = fix required before continuing.
rval validate <file>
2
File Watcher Daemon
Watches the project directory continuously. Validates every new or modified file in real time.
rval watch --project-dir .
3
Git Pre-commit Hook
Installed automatically. Validates all staged files before any commit is allowed. Cannot be removed by the agent.
rval hook install
4
CI/CD Gate
Pull requests to main are blocked until all specs pass quality thresholds and zero critical security violations remain.
rval gate --format github
